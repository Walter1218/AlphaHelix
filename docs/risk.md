# AlphaHelix 风险与边界

## 0. 核心约束、规则与纪律清单

以下约束是 AlphaHelix 的「红线」，任何代码、prompt 或流程改动都必须遵守。违反其中任何一条都可能导致回测失真、合规风险或系统不可维护。

### 0.1 数据与防穿越纪律

| 编号 | 约束 | 说明 | 实现位置 |
|---|---|---|---|
| C01 | **无未来函数** | T 日选股只能用 T 日及之前已公开的数据 | `agents.md` 3.6 节 |
| C02 | **财报按公告日过滤** | `fina_indicator` 必须满足 `ann_date <= trade_date` | `screen.py:fetch_fina_factors` |
| C03 | **历史 ST/退市过滤** | 用 `namechange` 历史名称判断，禁止用当前 `name` | `_tushare_utils.py:is_st_historical` |
| C04 | **退市/停牌校验** | `trade_date` 当天必须在 `daily` 中有交易记录 | `screen.py:pass1_screen` |
| C05 | **行业分类仅报告用** | `stock_basic` 的 `industry` 只做展示，不做历史量化截断 | `screen.py:build_universe` |
| C06 | **指数数据用对接口** | 个股用 `daily`，指数用 `index_daily` | `evaluate.py:get_close_price` |
| C07 | **只使用公开数据源** | 禁止接入非公开渠道或内幕信息 | 数据源仅限 Tushare |
| C38 | **策略决策必须 walk-forward / 样本外** | 任何影响选股决策的模型、因子、权重、阈值、参数，只能用决策点之前的数据生成；禁止在全样本上调优后回测同一区间 | `agents.md` 2.5 节 |

### 0.2 计算与评估纪律

| 编号 | 约束 | 说明 | 实现位置 |
|---|---|---|---|
| C08 | **LLM 不做数值计算** | 所有因子、收益、指标由 Python 计算 | `architecture.md` 2.1 节 |
| C09 | **评估必须确定性** | 命中率、超额收益等只能由 `evaluate.py` 计算，不得让 LLM 评估 | `evaluate.py` |
| C10 | **入场价用 T 日收盘价** | 回测买入价必须是 `trade_date` 收盘价 | `evaluate.py:evaluate` |
| C11 | **出场价用 T+H 日收盘价** | 回测卖出价必须是 `exit_date` 收盘价 | `evaluate.py:evaluate` |
| C12 | **每次改动后重跑 walk-forward** | 策略/因子/权重变更必须经完整回测验证 | `system-design.md` 5.2 |
| C13 | **保留 out-of-sample 区间** | 训练权重与验证收益不能使用同一段数据 | `improvement-plan.md` 7.6 节 |
| C14 | **回测需扣除交易成本** | 正式评估应包含印花税、佣金、滑点 | 待实现 |

| 编号 | 规则 | 阈值 | 实现位置 |

## 1. 数据层风险

### 1.1 Tushare 积分与限流

**风险**：高级接口需要积分，全量扫描容易触发限流。  
**缓解**：
- 本地缓存日线数据，增量更新
- 财务数据按季度更新
- 对 `screen.py` 加入请求间隔与重试
- 优先使用免费接口做 MVP

### 1.2 数据延迟

**风险**：日线数据通常为 T-1，盘中决策存在滞后。  
**缓解**：
- 明确选股时间点为收盘后
- 避免盘中实时交易决策

### 1.3 数据质量

**风险**：除权除息、停牌、复权处理不当导致因子失真。  
**缓解**：
- 使用 tushare 的复权接口或自行处理复权
- 停牌股票在计算动量时剔除

## 2. 模型层风险

### 2.1 LLM 数值幻觉

**风险**：LLM 可能算错收益率、PE 等数值。  
**缓解**：
- 所有数值计算交给 `scripts/screen.py`
- LLM 只负责解释与决策

### 2.2 过度自信

**风险**：LLM 给出高置信度但结果错误。  
**缓解**：
- 跟踪「置信度-实际收益」相关性
- 对置信度进行校准

### 2.3 定性信息误导

**风险**：新闻、研报情绪可能使模型追涨杀跌。  
**缓解**：
- 结合资金流与基本面交叉验证
- Cardinal 规则限制热点过度集中

## 3. 策略层风险

### 3.1 过拟合

**风险**：根据历史数据优化的因子在未来失效。  
**缓解**：
- 使用滚动窗口 walk-forward 回测
- 保留样本外测试期
- 定期重训练因子权重

### 3.2 未来函数

**风险**：决策时使用了未来才可知的信息。  
**缓解**：
- 严格执行：T 日决策只能用 T-1 及之前的数据
- 财务数据以 `ann_date` 为准
- 评估时不能用 T+1 之后的价格影响 T 日决策

### 3.3 市场风格切换

**风险**：动量/价值等因子在不同时期表现差异大。  
**缓解**：
- 多策略 ensemble
- Memory RAG 检索相似市场风格

## 4. 执行层风险

### 4.1 推荐高风险标的

**风险**：推荐 ST、退市、低流动性股票。  
**缓解**：
- Cardinal 规则拦截
- `screen.py` 初筛剔除
- RiskAgent 最终检查

### 4.2 行业过度集中

**风险**：组合集中在单一行业，暴露过大。  
**缓解**：
- Cardinal 规则：单行业占比 > 40% 警告
- 强制分散到 3-5 个行业

### 4.3 缺少止损

**风险**：推荐股票没有止损价。  
**缓解**：
- 输出 schema 强制包含 `stop_loss`
- 缺少时 Cardinal pause

## 5. 合规与伦理风险

### 5.1 投资建议误解

**风险**：用户将研究结果误认为是投资建议。  
**缓解**：
- 每次输出包含责任声明：说明我们对研究方法和数据质量负责，但不承诺收益
- README 与 SKILL 中明确研究性质与责任边界

### 5.2 内幕信息

**风险**：模型可能基于公开新闻推断未公开信息。  
**缓解**：
- 仅使用公开数据源
- 避免使用非公开渠道信息

### 5.3 算法交易合规

**风险**：若接入实盘交易，需符合监管要求。  
**缓解**：
- 当前阶段仅做研究，不接入实盘
- 实盘前咨询合规意见

## 6. 系统层风险

### 6.1 HelixAgent 版本变更

**风险**：HelixAgent 更新导致 plugin/tool API 不兼容。  
**缓解**：
- 使用目录扫描工具（稳定性高于 plugin API）
- 固定 HelixAgent 版本测试
- 保留回归测试

### 6.2 Token 成本失控

**风险**：每次选股调用多次 LLM，成本不可控。  
**缓解**：
- 因子计算本地化
- 限制候选股数量
- 设置每日调用预算

## 7. 边界声明

AlphaHelix 当前边界：
- 仅研究 A 股市场
- 仅做收盘后选股研究
- 不提供实时交易信号
- 不保证收益
- 优化目标为提升预测准确率，而非绝对收益

## 8. 责任声明

> AlphaHelix 对研究方法、数据来源、回测过程与因子逻辑的严谨性负责，并通过 walk-forward 与 Feedback Harness 持续优化模型。但证券市场受宏观环境、政策变化、市场情绪等不可控因素影响，模型输出不代表对未来收益的保证。使用者应结合自身判断审慎决策，过往表现不代表未来收益。
